# Validation Report: Reimbursement Forensics (ReiFor) v1

This report documents what was validated for the private 30-item sample generated by `generator.py`.

## What the benchmark measures

ReiFor measures **multi-document auditing under explicit policy constraints**:
extracting semi-structured receipt lines, applying a policy with interacting caps and exceptions, and reconciling across receipts and an email approval thread.

## Determinism and grading

- The gold for each item is a single **integer USD cents**.
- Scoring is exact-match on the integer string.
- The generator uses a seed, but the benchmark does **not** rely on a hidden seed for solvability: each answer is implied by the solver-visible documents.

## External solvability / identifiability argument

For each item `id`, the solver bundle includes:

- `common/policy.md`: the complete policy excerpt used for adjudication.
- `common/exchange_rates.csv`: the complete table mapping (date, currency) to USD/unit for conversion.
- `cases/<id>/receipts.txt`: all receipt lines with explicit fields (or explicit `?` missing markers).
- `cases/<id>/emails.txt`: an email thread that may include explicit approval directives in a fixed syntax.

A qualified external solver (human specialist or another model) can determine the correct reimbursable total because:

1. Every reimbursable/invalidating condition referenced in the rules is observable in the documents (receipt fields, missing markers, flags, approvals).
2. Currency conversion uses a solver-visible rate table indexed by receipt date and currency.
3. All caps are stated numerically and operate only on quantities visible in the receipts (per-receipt or per-day by receipt date).
4. The approval syntax is explicit and uniquely identifies which receipt is overridden and whether the override is FULL or PARTIAL to a specific cents amount.

Therefore the answer is **identifiable from public evidence** without access to any hidden generator state.

## What was verified (in this package)

- `verifier.py` recomputes each item’s gold **using only solver-bundle assets** and checks it matches `gold_private_sample.jsonl`.
- `scorer.py` produces an exact-match report.
- A weak baseline is included (`baseline_solver.py`) that naively sums converted receipt totals while ignoring key rules; on the current 30-item sample it scored **1/30**.

## Solver-bundle leakage check

The solver bundle contains only:

- `SOLVER_MANIFEST.json`
- `items_private_sample.jsonl`
- `README.md`
- `common/` assets
- `cases/` assets

It does **not** contain: gold answers, generator/verifier/scorer code, validation reports, or private traces.
